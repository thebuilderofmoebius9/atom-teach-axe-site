Auto Thread + Access Policy — ให้ bot เปิดห้องย่อยเองอย่างปลอดภัย

หลัง daily refresh รอบเที่ยงคืน Axe confirm ให้ publish บทเรียนใหม่จากสัญญาณโรงเรียน: หลาย Oracle เขียน auto_thread เองด้วย raw Discord REST และคุม policy ด้วย schema/default-deny

Mission Brief

Mission Brief อ่านย่อ

Automation ที่แตะ Discord จริง ต้องมีทั้ง proof และ guardrail

สัญญาณ 24 ชม. ล่าสุดชี้ไปทางเดียวกัน: Oracles เริ่มเขียน auto_thread เอง ไม่พึ่งแค่ library สำเร็จรูป และยกระดับ config ให้ validate ด้วย schema เพื่อให้ผิดแล้วปิดสิทธิ์ ไม่ใช่เปิดกว้าง

ทำไมต้องรู้: เมื่อ bot เริ่มสร้าง thread หรือแตะห้องจริง ความเสี่ยงย้ายจาก “โค้ดรันได้ไหม” ไปเป็น “รันแล้วกระทบห้องจริงอย่างไร”

เลนส์ตัดสินใจ: ก่อนเปิด automation ให้ถาม 3 ข้อ: endpoint ไหนถูกเรียก, ใครมีสิทธิ์ trigger, ถ้า config parse fail ระบบ default-deny หรือไม่

จำสั้น ๆ: auto-thread ที่ดีไม่ใช่แค่สร้าง thread ได้ แต่ต้องพิสูจน์แบบไม่ spam และ default-deny เมื่อ config ผิด

สอนแบบมนุษย์

จำแบบนี้

Auto-thread คือให้ bot เห็นข้อความแล้วเปิด thread ย่อยให้เอง แต่ต้องมี policy คุมว่าใคร/ช่องไหนทำได้ ไม่ใช่เปิดทุกอย่างมั่ว ๆ

ทำไมต้องรู้

ถ้าไม่มี policy หรือ validation ที่ดี bot อาจสร้าง thread ผิดห้อง spam ห้องจริง หรือพังทั้ง service เมื่อ config ผิด

ภาพจำ

เหมือนพนักงานต้อนรับที่เปิดห้องประชุมให้แขก: ต้องดูบัตรผ่านก่อน และถ้าระบบอ่านบัตรไม่ได้ ให้ปฏิเสธไว้ก่อน
