MCP ใน Session คือ Notification Queue ไม่ใช่ Tool Call

MCP (Model Context Protocol) ใน session run คือ asynchronous notification queue ไม่ใช่ synchronous tool call ซึ่งเปลี่ยนวิธีคิด session lifecycle และการอยู่รอด error handling

Mission Brief

Mission Brief อ่านย่อ

ความหลากหลายระหว่าง queue notification vs synchronous tool call เปลี่ยนวิธีออกแบบ MCP protocol

ใจความ: MCP ใน session คือ async queue ไม่ใช่ sync tool call — ประเด็นนี้ไม่ใช่งานปฏิบัติการเล็ก แต่เป็นวิธีมองระบบ: ต้องรู้ว่า session lifecycle ของเรา vs MCP message lifecycle เป็นคนละตน

ทำไมต้องรู้: สำคัญเพราะ ถ้าสับสนสองอย่างนี้ code จะ deadlock: chอว response ที่ไม่มา หรือ assume MCP message มาตรงเวลา ถ้าเข้าใจผิด ทีมจะใช้เวลาหาสาเหตุ timeout ที่บอกไม่ชัด

เลนส์ตัดสินใจ: มองผ่านเลนส์ผู้บริหาร: ถ้าออกแบบระบบที่ใช้ MCP ต้องตอบได้ว่า notification queue lifecycle กับ session lifecycle เราคิดเป็นอันเดียวหรือแยก — คำตอบเปลี่ยนการ allocate resource

จำสั้น ๆ: MCP ใน session คือ async queue ไม่ใช่ sync tool call ใช้ timeout + heartbeat เพื่อจัดการกรณีที่ queue ขาด

จำแบบนี้

MCP ใน session run คือ asynchronous notification queue ไม่ใช่ synchronous tool call — session จะปิดได้ แต่ MCP ยังรับข้อมูลต่อจากอื่น

ทำไมต้องรู้

ถ้าสับสนสองอย่างนี้ code จะ deadlock: chờ response ที่มาจาก queue เพราะคิดว่าต้อง synchronous จำขาด timeout logic หรือ assume MCP callback มาตรงเวลา

ภาพจำ

tool call เหมือนสั่งอาหาร: 'ฉันต้องการ X' → เชฟทำ → นำมา | MCP queue เหมือน inbox: ได้ข้อความจากคนหลาย ๆ คน ลำดับไม่แน่ session ของเราปิดแล้ว แต่ข้อความยังไหลเข้ามา

Practice

ลองทำก่อนดูเฉลย

จินตนาการ: session เปิด → stdin ไปถึง MCP → MCP subscribe → session task ปิดไป (สำเร็จ) → ยัง stdout รับ notifications → session ผู้ใช้จะ consume messages จากข้อมูลเก่า
